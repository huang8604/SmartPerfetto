# v2.1 Phase 3 — SDK-aware Active Compact + Bounded Recovery (Design Note)

> **Status**: design only, no code lands in this commit
> **Why doc-first**: Phase 3 modifies the `claudeRuntime` main loop and
> introduces SDK `interrupt()` / `resume` orchestration. Codex review F.5
> already flagged that the SDK has no mid-stream push API, so the design
> changed from the v2.1 plan's original "push recovery on
> compact_boundary" to a fallback shape. Locking that fallback in a
> design note before touching the runtime keeps the implementation PR
> small and review-friendly.

## What we already have

- **`compact_boundary` handler** in `claudeRuntime.ts:504-508` — sets a
  flag when the SDK auto-compacts; pushes a recovery note **after** the
  current analysis completes (`claudeRuntime.ts:815-879`).
- **Priority-ordered recovery note** with budget cap (`tryAdd()`,
  ~800-char ceiling). Sections: plan progress → next phase → findings
  → entity context.
- **`sdkQueryWithRetry`** (`claudeRuntime.ts:943-979`) — already wraps
  the SDK with `resume: sdkSessionId` so a new query can pick up the
  prior conversation. Used today for correction retries.
- **`AgentMetricsCollector`** records per-turn `inputTokens`,
  `cacheReadInputTokens`, `cacheCreationInputTokens`. The Phase 1.4 lite
  cache stability tests guarantee the prompt prefix is byte-stable, so
  the SDK's implicit cache works in our favour without explicit TTL.

## What this phase changes

### 3.1 — Token meter & pre-rot threshold

Add a per-turn check in the SDK stream loop (right after the `result`
message is consumed):

```ts
const accumulated = metricsCollector.getCumulativeUncachedInputTokens();
const threshold = SDK_CONTEXT_LIMIT * Number(process.env.CLAUDE_PRECOMPACT_THRESHOLD ?? 0.6);
if (accumulated >= threshold && !preCompactTriggered) {
  preCompactTriggered = true;
  this.emitUpdate({
    type: 'progress',
    content: { phase: 'analyzing', message: '⚠️ 接近上下文上限，准备切换会话承接' },
    timestamp: Date.now(),
  });
  // step 3.2 picks up here
}
```

`getCumulativeUncachedInputTokens()` is a derived value:
`inputTokens - cacheReadInputTokens - cacheCreationInputTokens`
(Phase 1.3 surfaces this via baseline aggregator already; the
collector itself just needs a getter).

Threshold default `0.6` of `SDK_CONTEXT_LIMIT` (200K for Sonnet 4.6 →
120K uncached input). Env override: `CLAUDE_PRECOMPACT_THRESHOLD`.

### 3.2 — Interrupt + resume + recovery preamble (the F.5 fallback)

The SDK's Query handle exposes `interrupt(): Promise<void>` (no message
push). The flow is:

```ts
1. preCompactTriggered → call query.interrupt()
2. Wait for the current async iterator to settle (already happens
   naturally — the loop's `for await` exits when interrupt resolves).
3. Build a recovery preamble using existing buildRecoveryNote() logic.
4. Start a new query with:
   - resume: sdkSessionId   // continue the same conversation
   - prompt: recoveryPreamble + originalUserQuery
5. Carry on streaming from the new query.
```

The recovery preamble is **prepended** to whatever message would
normally be the next user turn. The SDK `resume` keeps the prior
assistant context cached; the preamble takes the place of "what we did
so far" so the post-interrupt conversation does not start from zero.

### 3.3 — Bounded raw preservation in the recovery note

Today's `buildRecoveryNote()` only persists structured findings + plan
progress. After interrupt the agent loses the immediate-context tool
results. Patch the note builder to additionally include the **last
N=5 raw tool origins** in compact form:

```
Recent tool calls (last 5, raw form):
  1. mcp__smartperfetto__execute_sql — SELECT * FROM frame WHERE … (paramsHash:abc12345, 200 rows)
  2. mcp__smartperfetto__invoke_skill — startup_slow_reasons(traceId,appPackage) (paramsHash:def67890, 12 findings)
  …
```

We do **not** dump the full payloads — that would defeat the compaction
goal. We dump `{toolName, inputSummary, paramsHash, rowCount/findingCount}`
(all already on `ToolCallRecord` after Phase 0.5).

Budget for the raw preservation block: 200-300 chars; it slots between
"plan progress" and "findings" in the priority order so it survives
even when the budget tightens.

### 3.4 — Recitation rate-limiting

Codex review A.5 noted that injecting a phase-hint reminder into every
data-tool response is noise for short SmartPerfetto sessions (typical
20-25 turns). Restrict to:

- **Phase transition** — already happens in `update_plan_phase`.
- **Post-compact first tool result** — set a `justRecovered` flag in
  the runtime that the next data-tool response wrapper consults and
  prepends `next_phase_reminder` once.
- **High-risk tools** — `fetch_artifact(level=full|rows)` only;
  small-payload tools (`execute_sql` short results, `lookup_*`) skip.

This is independent of 3.1-3.3 and could ship in its own commit if
that's safer.

## Test plan

| What | How |
|------|-----|
| Token meter logic | Pure unit test on a `getCumulativeUncachedInputTokens()` helper — feed synthetic per-turn metrics, assert threshold trip |
| Interrupt + resume contract | Mock SDK `query()` returning a controllable async iterator; assert that on threshold, `interrupt()` is awaited and a second `sdkQuery({ resume })` is issued with the recovery preamble in `prompt` |
| Recovery note shape | Existing `buildRecoveryNote()` test extended with `recentToolCalls` fixture |
| Recitation rate-limit | Wrapper test that tracks how many tool responses carry `next_phase_reminder` over a 20-call sequence |
| End-to-end | Long-trace e2e (`scroll-demo-customer-scroll.pftrace` × heavier query). Assert (a) at least one progress event with "上下文上限" in long runs, (b) plan + findings survive across the interrupt boundary |

Compact recovery unit tests are the missing 5th group from Phase 4 —
they will land alongside this phase, not before.

## Implementation PR breakdown (suggested)

1. **3-1**: token meter + getter on `AgentMetricsCollector`. No SDK
   touching. Pure addition.
2. **3-2**: recovery note bounded raw preservation. Patches
   `buildRecoveryNote()` only. Independent of 3.1.
3. **3-3**: interrupt + resume orchestration in `claudeRuntime`. The
   risky one — should sit alone for review.
4. **3-4**: recitation rate-limit wrapper. Touches MCP wrappers; can
   ship after 3-3 once the post-compact `justRecovered` flag exists.

## Risks

- **Race between `interrupt()` and in-flight tool calls** — the SDK
  doesn't document partial-tool-call behaviour during interrupt. Need
  to verify via PoC that an interrupted tool call cleanly errors out
  rather than corrupting state.
- **Resume's actual cache behaviour** — `resume: sdkSessionId` reuses
  the conversation but we don't know whether the SDK re-sends the
  whole prior history or trusts its server-side cache. Bench the
  delta in `cache_read_input_tokens` before vs after the interrupt.
- **Phase 3.2 might double-count tool calls** in `toolCallHistory` if
  the watchdog/circuit breaker ticks during the interrupt window.
  Add a short cooldown (`interruptInProgress` flag) so those checks
  pause until resume completes.
